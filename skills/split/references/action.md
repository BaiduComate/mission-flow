职责：将拆分结果转换为 iCafe 卡片创建请求，调用 `icafe` skill（通过 `icafe-cli` 命令行工具）执行创建。

## 卡片字段映射

| 文档元素 | iCafe 字段 | 说明 |
|---------|-----------|------|
| 任务名称 | title | 格式：【端】【模块】功能描述，20-40 字 |
| 功能描述 | description | HTML 格式，Feature 和 Story 使用不同模板 |
| 父卡片 | card_type | Feature |
| 子卡片 | card_type | Story |
| Story 下任务 | card_type | Task |
| 负责人 | assignee_id | 默认使用触发操作的用户 |
| 状态 | status | 默认"新建" |
| 优先级 | priority | P0-Highest / P1-High / P2-Middle / P3-Low / P4-Lowest |

## Feature 描述模板

创建 Feature 卡片时 `description` 字段需为 HTML 格式，将以下 Markdown 模板转换后传入：

```markdown
## 需求概述
[模型对 PRD 的结构化总结，3-5 句话，回答"做什么"和"为什么做"]

## 原始信息来源
- PRD: [知识库链接]
- 补充材料: [其他链接，如有；若无则删除此条]

## 代码库现状

### 技术栈
- 前端: [框架及版本]
- 服务端: [语言 + 框架]
- 数据库: [类型]
- 其他: [如 Redis、Kafka 等；若无则删除此条]

### 模块映射
| 模块 | 路径 | 与需求的关联 | 现状 |
|------|------|-------------|------|
| [模块名] | [路径] | [关联说明] | 已存在，需改造 / 不存在，需新增 |

### 关键发现
- [影响全局实现的重要发现]

### 不确定项
- [需后续确认的事项；若无则删除此节]

## 任务总览
共 N 个 Story（前端 X 个，服务端 Y 个）
依赖链: [Story 间的依赖关系描述；若无则写"无"]
```

**数据来源：**
- "需求概述" ← 第一步文档分析结果
- "代码库现状" ← 第二步代码库调研报告（直接搬入）
- "任务总览" ← 第四步拆分结果汇总

## Story 描述模板

创建 Story 卡片时 `description` 字段需为 HTML 格式，将以下 Markdown 模板转换后传入：

```markdown
## 背景
[1-2 句话说明该 Story 在整体需求中的位置]
父需求: [Feature 卡片标题 + iCafe 链接]

## 目标
[该 Story 完成后，用户/系统能做什么，不超过 3 句]

## 详细说明
[具体功能描述、业务规则、边界条件]

## 关注文件与目录
| 路径 | 操作 | 说明 |
|------|------|------|
| [文件/目录路径] | 改造 / 新增 | [具体改动说明] |

## 技术要点
- [该 Story 特有的实现约束或建议]

## 依赖
- 依赖 Story [序号] [标题]: [具体依赖什么]
- 外部依赖: [如有]
[若无依赖则删除此节]

## 参考信息
- PRD 相关段落: [Feature 卡片中 PRD 链接 + 章节名]
- 相关文档: [API 文档、设计稿等链接；若无则删除此条]
```

**数据来源：**
- "关注文件与目录" ← 第二步调研报告的模块映射，按 Story 粒度拆分
- "技术要点" ← 第一步文档分析中与该 Story 相关的技术约束
- "依赖" ← 第四步合并判断中识别出的依赖关系
- "背景"中的父需求链接 ← 第六步创建 Feature 后获得的 iCafe 链接
