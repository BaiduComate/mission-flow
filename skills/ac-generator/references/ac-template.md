# 测试验证点格式规范与示例

## 核心原则

每条验证点必须满足 4 个标准：
1. **具体明确**：包含清晰的触发条件和预期结果
2. **可验证**：能够通过测试明确判断通过或失败
3. **可自动化**：能够通过脚本或 UI 自动化工具验证
4. **无二义性**：不同人理解的含义一致

## 输出格式

```markdown
## 功能验证点

1. 用户输入正确的用户名和密码后点击登录，页面跳转到首页并显示用户名和头像
2. 用户名为空时点击登录，输入框下方显示红色提示"请输入用户名"
3. ...
```

使用自然语言描述，句式为「[触发条件]，[预期结果]」，不使用 checkbox 格式。

## 数量控制

- 小需求：3-5 个
- 中需求：5-10 个
- 大需求：最多 10 个

合并同类验证点（如多个必填字段的空值校验合并为一条）。

## ✅ 好的示例

```
1. 用户输入正确的用户名和密码后点击登录，页面跳转到首页并显示用户名和头像
2. 用户名为空时点击登录，输入框下方显示红色提示"请输入用户名"
3. 用户名或密码错误时，显示"用户名或密码错误"，不跳转页面
4. 调用 GET /api/v1/orders 传入 page_size=150 时，返回 400 错误和提示"每页数量不能超过100"
5. 点击"导出"按钮，浏览器下载名为"订单列表_YYYYMMDD.xlsx"的文件
6. 筛选结果超过 10000 条时，显示"当前数据超过10000条，请缩小筛选范围后导出"，不执行导出
```

## ❌ 禁止的表述

- "登录功能正常" → 模糊，无法验证
- "性能良好" / "加载流畅" → 主观，无法量化
- "界面美观" → 主观判断
- "索引创建成功" / "表结构正确" → 基础设施，非业务验收

## 🚫 明确排除的验证点类型

1. **性能验证**：响应时间、并发量、QPS
2. **安全验证**：加密算法、渗透测试
3. **压力测试**：大数据量、高并发
4. **基础设施**：数据库索引、表结构、字段类型
5. **主观判断**：界面美观、体验流畅
6. **浏览器兼容性**：多浏览器、响应式布局、跨操作系统 UI
