# 需求拆分助手

**Announce at start:** "我正在使用 ai-prd-split skill 来拆分文档。"

## 依赖技能

- `icafe-official`：创建 iCafe 卡片及绑定父子关系
- `ku-doc-manage`：读取知识库文档

## 流程

```
读取并理解文档 → 调研代码库 → 【澄清歧义】 → 拆分任务 → 用户确认 → 创建卡片
```

### 第一步：读取并理解文档

判断文档来源类型，**必须**使用子 Agent 读取并提炼：

| 来源                                         | SOURCE_TYPE |
| -------------------------------------------- | ----------- |
| 知识库 URL（`https://ku.baidu-int.com/...`） | `ku-url`    |
| 本地文件路径                                 | `file`      |
| 其他 URL                                     | `url`       |

**特殊情况：** 用户直接粘贴文本时，不派发子 Agent，主 Agent 内联处理。

**派发子 Agent：**

使用 `references/document-reader.md` 作为 prompt，填入：

- `{{DOC_SOURCE}}`：文档 URL 或文件路径
- `{{SOURCE_TYPE}}`：上表中对应的来源类型

等待子 Agent 返回**文档读取报告**后：

- 若状态为 **DONE**：将文档类型、需求概述、功能模块表、业务规则带入后续步骤
- 若状态为 **FAILED**：告知用户失败原因，终止流程，不继续推进

### 第二步：调研代码库（子 Agent）

优先使用当前工作区作为代码库，只在无法对应时才询问用户。如果用户不提供代码库地址则跳过此步骤

**判断流程：**

```
检查当前工作区是否为代码库（存在 package.json / go.mod / pom.xml 等）
  → 是：直接派发子 Agent 调研
  → 否：检查是否有明显无关迹象（如当前目录是纯文档目录）
       → 无关：询问用户提供代码库路径
       → 不确定：先调研，若子 Agent 报告"未找到相关模块"，再询问用户
```

**派发子 Agent：**

使用 `references/codebase-explorer.md` 作为 prompt，填入：

- `{{PRD_SUMMARY}}`：第一步提取的需求摘要（功能模块 + 分工）
- `{{REPO_ROOT}}`：当前工作区路径，或用户提供的路径

等待子 Agent 返回**代码库调研报告**后：

- 若状态为 **DONE**：将相关模块表、技术栈、不确定项带入后续步骤
- 若状态为 **NEEDS_CONTEXT**：告知用户子 Agent 遇到的问题，询问补充信息，再决定是否重新派发或跳过本步骤

**跳过条件：** 用户明确说明是新项目，跳过本步骤，拆分时注明"均为新增模块"。

### 第三步：澄清歧义（按需，含代码库不确定项）

文档描述不明确，或代码库调研存在不确定项时，**必须**使用 question 工具询问用户。

**触发澄清的场景：**

- 功能边界不清晰
- 无法判断是前端还是服务端任务
- 文档提及功能但未说明具体行为
- 代码库调研报告中标注的不确定项

**严禁：** 基于"常识"或"最佳实践"推断未在文档中明确描述的功能（如配置页面、管理后台、监控报表等）。

### 第四步：拆分任务

**每张卡必须满足：**

1. **有明确交付物** — 能被演示或验收的功能，而不是"开发某模块"这类描述
2. **可独立开发** — 不依赖其他卡的代码才能启动
3. **单一职责** — 一个端、一条业务链路，不跨端、不跨链路
4. **来自代码库调研** — 结合第二步的调研报告，在描述中明确标注"改造现有模块"或"新增模块"
5. **标注关注文件** — 结合第二步调研报告的模块映射，为每张 Story 列出具体的关注文件/目录及操作类型（改造/新增），写入 Story 描述的"关注文件与目录"段

**拆分策略：**

- 前端：跨端必拆（前端页面、客户端 SDK、客户端（iOS/Android/Mac/Windows）、 桌面端(Mac/Windows)、 移动端(iOS/Android)），明确不做的端无须建卡
- 服务端：后端接口、微服务、网关、中间件、数据库设计
- 同端内按业务链路合并，禁止跨端合并为一张卡
- 不为 UI/UE、QA、运维、产品分配任务

**合并判断（按优先级检查）：**

| 判断条件                                | 动作     | 理由                                                    |
| --------------------------------------- | -------- | ------------------------------------------------------- |
| 同一业务链路的连续步骤                  | 合并     | 保持业务完整性，便于验收                                |
| 存在代码依赖（A 依赖 B 的代码才能运行） | 合并     | 保证可独立开发                                          |
| 拆分后粒度过小（< 1 人日）              | 合并     | 合并后的工时不可 > 5 人日，避免卡片碎片化，降低管理开销 |
| 技术栈不同且可独立交付                  | 保持分开 | 便于不同角色并行开发                                    |
| 跨端（前端/服务端）                     | 保持分开 | 端是任务拆分的最小边界，禁止跨端合并                    |

**严谨推断** — 只拆文档中明确提到的内容，不推断未描述的功能

**self-review（拆分完成后内联执行）：**

- [ ] 每张卡是否有明确的可验收交付物？
- [ ] 是否有卡依赖其他卡的代码才能启动？
- [ ] 是否有跨端任务（一张卡同时包含前端和服务端工作）？
- [ ] 是否有文档未明确提及的推断任务？
- [ ] 标题是否符合格式规范？
- [ ] （结果检验）单张卡工作量是否明显过小（半天以内）或过大（超过 5 天）？若有则重新检查合并/拆分

**任务标题规范：**

语法：`【SDD】【<端名称>】【<模块/服务名>】<功能描述>`
示例：`【SDD】【服务端】【Bot Gateway】WebSocket 连接管理与协议编解码开发`

### 第五步：展示并等待用户确认

```
需求拆分完成

#### 需求概览
【1-2 句话描述核心目标】

#### 拆分思路
【说明关键的合并/拆分判断。例："XX 和 YY 合并为一张前端卡，因为同属同一业务链路；ZZ 独立成一张服务端卡，因为技术栈不同且可独立交付。" 若无特殊决策则省略此节。】

#### 前端任务
| 序号 | 卡片标题 | 卡片内容描述 | 负责人角色 |
|-----|---------|---------|-----------|
| 1 | 【前端】【模块A】功能开发 | 功能描述 | 前端开发 |

#### 服务端任务
| 序号 | 卡片标题 | 卡片内容描述 | 负责人角色 |
|-----|---------|---------|-----------|
| 1 | 【服务端】【模块A】功能开发 | 功能描述 | 服务端开发 |

---

下一步：创建 iCafe 卡片

为你推荐最近访问的 iCafe 空间：
- prefixCode（空间名称）

请确认目标 iCafe 空间后执行创建卡片任务
```

卡片超过 3 个时，在第三个后插入分割线。

**HARD-GATE：** 必须等待用户明确确认后才能创建卡片。用户取消或超时未响应，终止流程，不创建。若用户未指定 iCafe 空间，询问后再继续。

### 第六步：创建卡片

用户确认后，调用 `icafe-card-assistant`，参考 `references/action.md` 执行卡片创建。

- 创建卡片类型：1 个 Feature 类型和若干个 Story 的卡片
- Feature 卡片使用 Feature 描述模板，Story 卡片使用 Story 描述模板。

**Feature 卡片的数据来源：**

- "需求概述" ← 第一步文档分析结果以及文档链接
- "代码库现状" ← 第二步代码库调研报告（直接搬入整体结构、技术栈、模块映射、关键发现、不确定项）
- "任务总览" ← 第四步拆分**详细**结果汇总

**Story 卡片的数据来源：**

- "关注文件与目录" ← 第二步调研报告的模块映射，按 Story 粒度拆分
- "详细" ← 第二步调研报告的模块映射，按 Story 粒度拆分后的 需求切片
- "技术要点" ← 第一步文档分析中与该 Story 相关的技术约束
- "依赖" ← 第四步合并判断中识别出的依赖关系
- "背景"中的父需求链接 ← 创建 Feature 后获得的 iCafe 链接

**创建顺序：**

1. 先创建 Feature 父卡片，获取父卡片 ID
2. 等待 1 秒，确保父卡片完全导入到 iCafe 系统
3. 创建 Story 子卡片时通过 `parent` 参数同时绑定父卡片（不可先创建再 update 绑定）
4. 如遇 `NOT_IMPORTED` 错误，等待 1 秒后重试；仍失败则告知用户手动关联
5. 如遇 API 调用失败，提示用户检查 token 配置（环境变量 `COMATE_AUTH_TOKEN` 或 `~/.comate/login`）

## 禁止行为

- 跳过用户确认直接创建卡片
- 超时后擅自创建卡片
- 未经用户明确指定，创建非 Feature 类型父卡片
- 创建非 Story 类型子卡片
- 按功能模块（而非端）分组输出任务
- 将无独立交付物的任务单独成卡（如"开发某接口"、"调试某函数"）
- 推断文档未明确描述的需求场景
- 负责人不使用默认触发人
- 创建完子卡片后再单独调用 update 绑定父卡片

## 终端状态

卡片创建完成后，告知用户：

"需求拆分完成，所有信息已写入 iCafe 卡片（Feature 卡片包含完整的代码库现状和模块映射，Story 卡片包含关注文件与技术要点）。如需进行架构设计，可在新会话中对具体 Story 卡片触发 architecture-design skill。"
